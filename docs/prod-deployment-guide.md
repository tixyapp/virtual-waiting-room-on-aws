## Issues Found and Fixed During Deployment

This section documents every failure encountered during the first production deploy
and the fix applied. Each fix is already baked into the source files.

---

### Issue 1 — Artifact bucket naming convention

**What happened:** The `deploy.sh` and CloudFormation templates append `-$REGION` to the
base bucket name automatically (`s3://$BUCKET-$R/`). Creating the bucket as
`tixy-wvroom-prod-eu-west` meant the scripts would try to access
`tixy-wvroom-prod-eu-west-eu-west-1` (double region suffix) which doesn't exist.

**Fix:** Bucket must be named so that `{base}-{region}` equals the full name.
Correct base: `tixy-prod-wvroom` → full bucket name: `tixy-prod-wvroom-eu-west-1`.

**Rule:** Always check the bucket base name convention before creating the bucket:
```bash
# The build script and CFN templates derive the bucket as:
# {BUILD_BUCKET}-{REGION}
# So: BUILD_BUCKET=tixy-prod-wvroom → bucket=tixy-prod-wvroom-eu-west-1
```

### Issue 2 — `DetectAbandonedRole` IAM policy rejected empty `SessionEventsSnsArn`

**Stack event:**
```
Resource handler returned message: "Resource must be in ARN format or '*'."
Resource: DetectAbandonedRole
```

**Root cause:** The `DetectAbandonedRole` IAM policy has an `sns:Publish` statement
with `Resource: {"Ref": "SessionEventsSnsArn"}`. When the parameter is passed as
an empty string `""` (required for the first deploy before the inlet stack exists),
IAM rejects the empty string as an invalid ARN.

**Fix applied to `deployment/virtual-waiting-room-on-aws.json`:**
Added a `HasSnsArn` condition and changed the SNS resource to use `Fn::If` with
a scoped wildcard fallback:

```json
"Conditions": {
  "HasSnsArn": {
    "Fn::Not": [{"Fn::Equals": [{"Ref": "SessionEventsSnsArn"}, ""]}]
  }
}
```

```json
"Resource": {
  "Fn::If": [
    "HasSnsArn",
    {"Ref": "SessionEventsSnsArn"},
    {"Fn::Sub": "arn:aws:sns:${AWS::Region}:${AWS::AccountId}:*"}
  ]
}
```

This allows the main stack to be created with an empty `SessionEventsSnsArn` on
first deploy, and tightened to the real ARN in the follow-up update (Step 5).

### Issue 3 — Wrong template file uploaded to S3 (unresolved `%%` placeholders)

**Stack event:**
```
Transform AWS::Include failed with: The location parameter is not a valid S3 uri.
```

**Root cause:** We uploaded `deployment/virtual-waiting-room-on-aws.json` (the raw
source file) directly to S3 with `aws s3 cp`. This file still contains
`%%BUCKET_NAME%%`, `%%SOLUTION_NAME%%`, `%%VERSION%%`, `%%TIMESTAMP%%`
placeholders. The `AWS::Include` transform tried to resolve an S3 URI containing
literal `%%` strings, which is invalid.

**Fix:** Always upload the processed template from `deployment/global-s3-assets/`
(generated by `build-s3-dist.sh`), not the raw source file:

```bash
# WRONG — raw source, has unresolved placeholders
aws s3 cp deployment/virtual-waiting-room-on-aws.json s3://...

# CORRECT — processed output from build-s3-dist.sh
aws s3 cp deployment/global-s3-assets/virtual-waiting-room-on-aws.template s3://...
```

**Rule:** After patching a template mid-deploy, always apply the patch to
`deployment/global-s3-assets/virtual-waiting-room-on-aws.template` (the
already-built file), not the raw `.json` source. The raw `.json` is the
permanent source of truth; `global-s3-assets/` is the deployable artifact.

### Issue 4 — `StageDescription` rejected on `PublicWaitingRoomApiDeployment`

**Stack event:**
```
Resource handler returned message: "StageDescription cannot be specified when
stage referenced by StageName already exists"
Resource: PublicWaitingRoomApiDeployment
```

**Root cause:** The original `PublicWaitingRoomApiDeployment` resource combined
`StageName`, `StageDescription`, and `AccessLogSetting` in a single
`AWS::ApiGateway::Deployment` resource. When CloudFormation creates a REST API
from a Swagger body (via `AWS::Include`), API Gateway implicitly creates the
stage as part of the import. By the time the `Deployment` resource runs, the
stage `api` already exists, and setting `StageDescription` on an existing stage
via a Deployment is rejected.

This failure only appeared on a fresh `create-stack` (not on `update-stack`),
which is why it worked in dev (where the stack was originally deployed before
the Swagger import was added).

**Fix applied to both `deployment/virtual-waiting-room-on-aws.json` and
`deployment/global-s3-assets/virtual-waiting-room-on-aws.template`:**

1. Removed `StageName` and `StageDescription` from `PublicWaitingRoomApiDeployment`
2. Added a separate `AWS::ApiGateway::Stage` resource (`PublicWaitingRoomApiStage`)
   that references the deployment and sets logging/access log configuration
3. Updated `DependsOn` on `ApiKey` and `UsagePlan` to reference
   `PublicWaitingRoomApiStage` instead of `PublicWaitingRoomApiDeployment`

```json
"PublicWaitingRoomApiStage": {
  "Type": "AWS::ApiGateway::Stage",
  "DependsOn": ["PublicWaitingRoomApiDeployment"],
  "Properties": {
    "RestApiId": {"Ref": "PublicWaitingRoomApi"},
    "DeploymentId": {"Ref": "PublicWaitingRoomApiDeployment"},
    "StageName": {"Fn::FindInMap": ["CoreApi", "Stage", "Name"]},
    "AccessLogSetting": { ... },
    "MethodSettings": [{ "ResourcePath": "/*", "HttpMethod": "*", "LoggingLevel": "ERROR" }]
  }
}
```

### Issue 5 — API Gateway CloudWatch Logs role not configured in account

**Stack event:**
```
Resource handler returned message: "CloudWatch Logs role ARN must be set in
account settings to enable logging"
Resource: PublicWaitingRoomApiStage
```

**Root cause:** Enabling access logging on an API Gateway stage requires the
account-level CloudWatch Logs role ARN to be configured in API Gateway settings.
This is a one-time per-account setup. The `tixy-prod` account had never had a
waiting room stack deployed before, so this setting was missing.

**Fix:** Deploy the prerequisite stack first (included in the build artifacts):

```bash
AWS_PROFILE=tixy-prod aws cloudformation create-stack \
  --stack-name tixy-apigw-cw-role \
  --template-url "https://tixy-prod-wvroom-eu-west-1.s3-eu-west-1.amazonaws.com/virtual-waiting-room-on-aws/v1.1.12/virtual-waiting-room-on-aws-api-gateway-cw-logs-role.template" \
  --capabilities CAPABILITY_IAM \
  --region eu-west-1 --profile tixy-prod
```

**Rule:** On any fresh AWS account, always deploy `tixy-apigw-cw-role` stack
**before** the main waiting room stack.

## Correct First-Time Deploy Order

```
0. tixy-apigw-cw-role       ← account prerequisite (one-time per account)
1. main stack (SessionEventsSnsArn="")
2. inlet stack (uses PrivateApiInvokeURL from main stack outputs)
3. update main stack (add SessionEventsSnsArn from inlet stack output)
4. static site S3 bucket + CloudFront distribution
5. upload HTML pages to S3 under event prefix
6. deploy gate.js CloudFront Function, associate with static site distribution
```

## Static Site — Multi-Event Layout

One CloudFront distribution + one S3 bucket serves all events. Each event gets
its own S3 prefix. Any number of custom domains can CNAME to the same distribution.

```
s3://tixy-wvroom-prod-site/
  ye-poland-2026-04/
    queue/index.html       ← CONFIG.EVENT_ID = 'ye-poland-2026-04'
    buy-ticket/index.html
  other-event-2026-06/
    queue/index.html       ← CONFIG.EVENT_ID = 'other-event-2026-06'
    buy-ticket/index.html
```

To add a custom domain:
1. Request ACM certificate in `us-east-1` (CloudFront requirement) covering the domain
2. Add the domain as a CloudFront alternate domain name
3. Add a DNS ALIAS (Route 53) or CNAME record pointing at the CloudFront domain

## Sale Day Runbook

```bash
source .env.prod

# Before each new event (resets Redis state)
EVENT_ID=<new-event-id> ./scripts/new-event.sh

# ~10 min before tickets open
./scripts/pre-sale.sh           # enables PeriodicInletRule, sets INCREMENT_BY

# Monitor live
./scripts/queue-status.sh --watch 10

# Emergency: pause admission (disables periodic drip, MaxSizeInlet still runs)
RULE_NAME=$(aws cloudformation describe-stack-resource \
  --stack-name tixy-inlet-prod \
  --logical-resource-id PeriodicInletRule \
  --profile tixy-prod --region eu-west-1 \
  --query "StackResourceDetail.PhysicalResourceId" --output text)
AWS_PROFILE=tixy-prod aws events disable-rule --name "$RULE_NAME" --region eu-west-1

# After sale closes
./scripts/close-queue.sh
```
